03 — Two StoresIntermediate⏳ · 10 min read

Working Memory Thinks. Long-Term Memory Remembers.

At any moment your agent juggles two fundamentally different stores. The boundary between them is where the most interesting design happens.

On this page
The two storesThe handoffMemory lifecycleDeep dives
Overview

Two fundamentally different stores

✏️ Working Memory

The active context window. Everything the model sees right now. Fast. Small. Temporary.

  • Capacity: 1k–200k tokens
  • Access: zero latency — it's in the prompt
  • Lifespan: this session only
Deep dive →

💾 Long-Term Memory

External store (vector DB, KV, graph). Persists across sessions. Retrieved on demand.

  • Capacity: effectively unlimited
  • Access: milliseconds (query)
  • Lifespan: indefinite
Deep dive →

The handoff between these two stores is where most interesting engineering happens. What gets committed to long-term? When? At what granularity?

The Hard Part

The handoff decision

Not every sentence deserves a permanent memory. "What's the weather?" doesn't. "I have a severe nut allergy" absolutely does.

Agents use a secondary LLM call to make this decision. After each turn, a cheap fast model answers: "Does this turn contain facts worth keeping?"

write_decision.py
WRITE_PROMPT = """
Analyze this turn. Extract facts worth storing long-term.
Return JSON: {"worth_saving": bool, "facts": [str]}
Turn: {turn}
"""

def should_save(turn):
    result = llm.call(WRITE_PROMPT.format(turn=turn))
    return json.loads(result)

# "What's 2+2?" -> {"worth_saving": false, "facts": []}
# "I'm vegetarian" -> {"worth_saving": true, "facts": ["User is vegetarian"]}
💡 Granularity tip

Store extracted facts, not raw turns. "User is vegetarian" (4 tokens) beats the original sentence (15 tokens). Extraction improves both storage efficiency and retrieval precision.

Memory Lifecycle

Write → Age → Supersede → Redact → Forget

Memory isn't static. Facts age. Users change. A well-designed system handles the full lifecycle — including marking old facts as superseded (not deleted) so temporal queries still work.

See 03c — When writes conflict for a deep dive into supersession and governance.
