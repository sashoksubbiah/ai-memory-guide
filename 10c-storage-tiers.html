Hot / Warm / Cold: The Three Storage Tiers

Not every memory needs a vector lookup. A KV cache for the most-accessed facts pays for itself in latency and cost.

On this page
Why tier storageHot tierWarm tierCold tierRouting logicQuiz
Motivation

Why tier storage

Different memories have very different access patterns. The user's name and active project are accessed on every turn. A conversation from 18 months ago is accessed almost never. Storing both in the same place with the same latency and cost profile is wasteful.

💡 The fundamental insight

Access frequency determines tier. Design the tiers around observed access patterns, not assumptions.

Hot Tier

Sub-millisecond: KV cache

🔥 Hot tier — what lives here

  • Top 10–20 most-accessed facts per user
  • Current session state
  • Active project context
  • User preferences fetched every turn

📈 Characteristics

  • Read latency: <1ms
  • Storage: Redis or in-memory
  • Size: kilobytes per user
  • TTL: 15 min to 24 hours
  • Invalidated on write
hot.py
# Write to hot tier
r.setex(f"hot:{user_id}", 900, json.dumps(top_facts))

# Read from hot tier (sub-millisecond)
cached = r.get(f"hot:{user_id}")

# Invalidate when new fact written
r.delete(f"hot:{user_id}")
Warm Tier

Milliseconds: vector DB

🌡️ Warm tier — what lives here

  • Recent conversation summaries
  • Stated user preferences (all)
  • Session history (last 30–90 days)
  • Active project memories

📈 Characteristics

  • Read latency: 50–200ms
  • Storage: Qdrant / pgvector / Pinecone
  • Size: megabytes per user
  • No fixed TTL — evicted by inactivity
  • Semantic search enabled
Cold Tier

Seconds: archival storage

❄️ Cold tier — what lives here

  • Old episodic logs (>90 days)
  • Raw conversation transcripts
  • Archived memories evicted from warm
  • Compliance and audit logs

📈 Characteristics

  • Read latency: seconds
  • Storage: S3 / append-only log / Glacier
  • Size: gigabytes per account
  • Retained for compliance period
  • Accessed for backfill and audit only
Routing Logic

How memories move between tiers

1

Hot → Warm: TTL expiry

After TTL expires (15min–24h), the hot cache entry is gone. Next access falls through to the warm vector DB and re-populates hot on a cache miss.

2

Warm → Cold: inactivity eviction

After 30–90 days without access, old episodes are moved from the vector DB to S3. A metadata pointer remains in the warm store so they can be found later.

3

Cold → Warm: backfill on demand

When a query hits a cold-tier pointer, data is retrieved from S3, promoted back to warm, and optionally back to hot. This covers queries about distant history.

routing.py
def get_facts(user_id, query):
    # 1. Hot tier (sub-ms)
    hot = redis.get(f"hot:{user_id}")
    if hot:
        return json.loads(hot)

    # 2. Warm tier (vector search, ~100ms)
    warm = vector_db.search(query, user_id=user_id, top_k=5)
    if warm:
        redis.setex(f"hot:{user_id}", 900, json.dumps([m.text for m in warm]))
        return warm

    # 3. Cold tier (backfill from S3, seconds)
    cold = s3.fetch_archive(user_id)
    vector_db.upsert(cold)  # promote to warm
    return cold
