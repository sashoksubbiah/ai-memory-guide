Working Memory: The Live Scratchpad

Everything the model is actively reasoning over — current turns, tool results, retrieved facts. It's the model's desk.

On this page

Definition

What working memory is

Working memory is the active contents of the context window at any given moment. It's everything spread out on the model's desk right now. The model can see it and reason over it. When the session ends, it's gone.

💡 Mental model

Think of working memory like RAM — fast, small, and volatile. Long-term memory is the hard drive — slow, large, persistent. Agents need both.

What's Inside

historyRecent conversation turns (last 5–10, or as many as fit)

Capacity

Size and limits

Working memory is bounded by the context window. Modern models give 128k–200k tokens. A production system fills this with:

ComponentTypical sizeNotes
System prompt500–2,000 tokensInstructions, persona, tool definitions
Retrieved memories200–800 tokensTop 3–5 facts, each ~50–150 tokens
Conversation history500–5,000 tokensLast 5–20 turns
Tool results0–10,000 tokensCan be huge when returning documents
Current message10–500 tokensTypical user query
Chain of Thought

During reasoning

When a model generates a chain of thought — reasoning, planning, step-by-step problem solving — that reasoning is also working memory. The model's thinking is being written into the context window as it happens, and it can refer back to earlier reasoning steps as it writes later ones.

This is why larger context windows improve reasoning performance on hard tasks — the model has more desk space to spread out its work.

💡 Implication

For complex multi-step tasks, don't aggressively compress the prompt. Give the model room to reason. The cost of a slightly larger prompt is almost always worth the quality gain.
