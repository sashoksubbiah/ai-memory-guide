Multi-Agent Memory is a Graph, Not a Store

As soon as you have multiple agents, memory becomes a graph of scopes with permissions, ownership, and propagation rules.

On this page
Why it gets complexMemory topologyPrivate by defaultDeep divesQuiz
Motivation

Why multi-agent memory is hard

A single agent answers "what do I remember?" A team asks: which agent remembers which fact, on whose behalf, and who else can read it? Memory becomes a graph of stores with scopes, permissions, and propagation rules.

Topology

How stores are organized

A typical multi-agent system has several memory scopes:

ScopeWho readsWho writesExample content
PrivateOwning agent onlyOwning agent onlyResearch notes, internal reasoning
ProjectAll project agentsAll project agentsDecisions, architecture, task state
User profileAll agents for this userGoverned pipelineUser preferences, facts, history
OrganizationAll agents in orgDesignated writersPolicies, runbooks, procedures
💡 The topology should be explicit

Default "everyone can read everything" is almost never right. Explicit scope definitions contain damage when an agent makes mistakes.

The Most Important Rule

Private by default

Private memory by default, shared memory by explicit act. A research note the Researcher jots down for itself does not need to land in the project channel. Cross-scope writes should be deliberate, with a policy attached.

⚠ The sharing anti-pattern

Broad sharing surfaces mean any agent write error propagates everywhere. Narrow scopes contain damage.
