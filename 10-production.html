In Production, Memory is a System

Not just "install a vector DB" — a read path, a write path, background workers, multi-tenant isolation, latency budgets, and a plan for when things change.

Architecture

The full production memory system

Adding memory to a real product requires thinking through six concerns simultaneously: an API surface, separate read and write paths, background extraction workers, multi-tenant isolation, latency budgets, and an observability plan.

⚠ Memory is a product, not a feature

Memory engineering is retrieval engineering. Treat it like a first-class product with its own SLA, monitoring, and on-call rotation — not an afterthought added to the LLM call.

Two Paths

Separate read and write paths

The read and write paths have completely different performance requirements. The read path is on the user-facing latency budget. The write path is async and should never block a response.

1

User request arrives

Auth and scope check. Agent runtime is on the request path. Budget 800ms total.

2

Read path (sync)

Memory service: dense search + BM25 + graph walk + RRF fusion + rerank + permission filter + pack into context.

3

LLM generates

Response returned to user. Total user-perceived latency includes the full read path.

4

Write path (async)

Background worker: extract facts from the turn, PII filter, conflict check, upsert to vector store.

5

Background jobs

Summarization, re-embedding, decay, deduplication. Run offline — never on the request path.

💡 The critical rule

Never let a write failure block a user response. The read path must remain available even if the write path is degraded. Queue writes; retry on failure; alert on queue depth.

Background Workers

What runs offline

Several expensive operations must be pulled off the request path into background jobs:

JobWhat it doesFrequency
Fact extractorExtract structured facts from raw conversation turnsNear-realtime (stream consumer)
PII scannerScan and redact PII in existing memoriesNightly
Re-embedderRe-embed all memories when the embedding model changesOn model change
DeduplicatorFind and merge near-duplicate memoriesWeekly
Decay runnerReduce confidence of old unreinforced memoriesDaily
Review queueSurface low-confidence shared writes for human reviewDaily
⚠ Background jobs need the same reliability care as your API

A background job that silently fails is often worse than one that errors loudly. Monitor these jobs with alerts, dead-letter queues, and retry budgets.
