Rolling Summary: Compress and Carry Forward

Extend context without a vector database by periodically compressing old turns into a running summary.

On this page
What it isImplementationTradeoffsWhen to use itQuiz
Definition

What a rolling summary is

A rolling summary periodically compresses old conversation turns into a short paragraph using an LLM, then replaces the original turns with that summary. You extend effective context without needing external storage.

💡 The mental model

Imagine taking notes during a long lecture. Every 30 minutes you write a one-paragraph summary of what was covered, then close those pages. You still have the key facts, but they now take up far less space.

Implementation

How to build it

rolling_summary.py
class RollingSummaryBuffer:
    def __init__(self, max_turns=20, summary_every=10):
        self.summary = ""  # The compressed past
        self.recent = []   # Verbatim recent turns
        self.max_turns = max_turns
        self.summary_every = summary_every

    def add(self, role, content):
        self.recent.append((role, content))
        if len(self.recent) >= self.summary_every:
            self._compress()

    def _compress(self):
        # Compress oldest half of recent turns
        to_compress = self.recent[:self.summary_every // 2]
        chunk = "\n".join(f"{r}: {c}" for r, c in to_compress)
        new_summary = llm.call(
            f"Existing summary: {self.summary}\n\nNew conversation:\n{chunk}\n\n"
            f"Write a concise updated summary preserving all key facts."
        )
        self.summary = new_summary
        self.recent = self.recent[self.summary_every // 2:]

    def to_prompt(self):
        parts = []
        if self.summary:
            parts.append(f"[CONVERSATION SUMMARY]\n{self.summary}")
        parts += [f"{r}: {c}" for r, c in self.recent]
        return "\n".join(parts)
⚠ Summary LLM call adds latency

Each compression step requires an LLM call. Budget 200–500ms and a few cents per compression. Schedule it asynchronously when possible.

Tradeoffs

Comparison at a glance

AspectRolling Summary
External storageNone required
Semantic recallNo — summary is injected verbatim, not searched
Information lossYes — lossy compression
Latency overheadCompression call every N turns (~200–500ms)
Best forMedium-length sessions where verbatim detail is not critical
Fails atLong sessions, detailed technical context, multi-session continuity
When to Use It

Choosing rolling summary

✓ Use rolling summary when

You need more than a simple buffer but do not want to set up a vector database yet. Sessions are medium-length (20–100 turns). Losing exact wording of old turns is acceptable as long as key facts are preserved.

✗ Avoid when

Sessions are very long. Users reference exact past statements. Cross-session continuity is required. Any of these needs point to a vector store.

What information does rolling summary lose compared to storing verbatim turns?
ANothing — summaries are lossless
BSubtle details, exact wording, and nuance that the LLM chose not to include in the summary
COnly the timestamp of each turn
DAll semantic meaning
