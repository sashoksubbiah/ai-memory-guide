The Minimal Memory API

Three endpoints are enough to ship a complete memory system. Here is the minimum surface area and the auth pattern that keeps it safe.

Design Philosophy

Start with three endpoints

The minimum viable memory API has exactly three endpoints: ingest events, search memories, and delete a memory. Everything else — consolidation, supersession, analytics, summarization — is built on top of these three. Ship these first.

💡 YAGNI applies here

You are not going to need a 20-endpoint memory API on day one. Ship the three endpoints, get real usage data, then add complexity only where you see actual patterns.

Endpoint 1

POST /memory/events

This endpoint accepts raw conversation events and queues them for background processing. It does not write memories directly — that happens in a background worker.

ingest.py
class EventPayload(BaseModel):
    text: str          # raw turn text
    user_id: str       # who said it (validated against JWT)
    session_id: str    # which session
    metadata: dict = {}

@app.post("/memory/events")
async def ingest_event(event: EventPayload, user=Depends(auth)):
    # Never trust caller-supplied user_id
    if user.id != event.user_id:
        raise HTTPException(403)

    # Queue for async processing — never blocks the user response
    await queue.publish(event.model_dump())
    return {"status": "queued"}
⚠ Why queue instead of write directly

The extraction pipeline (PII check, fact extraction, embedding, deduplication) may take 500ms–2s. Queuing keeps the ingest endpoint at <20ms. It also allows retries on failure without affecting the caller.

Endpoint 2

POST /memory/search

This is the read path. Takes a query string and returns the top-k most relevant memories for the authenticated user.

search.py
class SearchPayload(BaseModel):
    query: str
    top_k: int = 5
    created_after: str | None = None   # optional temporal filter

@app.post("/memory/search")
async def search(payload: SearchPayload, user=Depends(auth)):
    results = await memory_service.search(
        query=payload.query,
        user_id=user.id,           # always from JWT, never caller-supplied
        top_k=payload.top_k,
        created_after=payload.created_after,
    )
    return {"results": [r.to_dict() for r in results], "count": len(results)}
⚠ user_id from JWT, always

Never accept user_id as a request body parameter for a read endpoint. Extract it from the verified JWT. One slip here is a cross-tenant data leak.

Endpoint 3

DELETE /memory/{id}

The forget endpoint. Deletes a single memory by ID, verifies ownership, propagates to derived indexes, and logs an audit event.

delete.py
@app.delete("/memory/{memory_id}")
async def forget(memory_id: str, user=Depends(auth)):
    memory = await memory_service.get(memory_id)
    if not memory:
        raise HTTPException(404)
    if memory.user_id != user.id:
        raise HTTPException(403)   # ownership check before deleting

    await memory_service.delete(memory_id)   # propagates to all indexes

    await audit_log.write({
        "action": "delete",
        "memory_id": memory_id,
        "actor": user.id,
        "timestamp": utcnow(),
    })
    return {"status": "deleted"}
Auth and Scoping

Enforcing isolation on every endpoint

1

JWT contains user_id and tenant_id

Verify the JWT on every request. Extract user_id and tenant_id from the verified token. Never trust these values from the request body.

2

Scope filter on every read

Every search call includes user_id as a mandatory filter. In multi-tenant systems, add tenant_id. The memory service layer enforces this, not the caller.

3

Ownership check before every delete

Verify that the memory being deleted belongs to the authenticated user. Do this before any deletion logic executes.

4

Audit log on every write and delete

Log every ingest acknowledgment and every deletion with actor, timestamp, and outcome. This is your audit trail for compliance.

Growth Path

What to build after the three endpoints

EndpointWhen to add
GET /memory/{id}When clients need to fetch a specific memory by ID for display
PATCH /memory/{id}When you want users to be able to correct stored memories
GET /memory/users/{user_id}/statsWhen you need per-user observability on memory size and access patterns
POST /memory/consolidateWhen you need deduplication and merging as an explicit background operation
