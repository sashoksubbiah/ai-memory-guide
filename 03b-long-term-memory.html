Long-Term Memory: Persistent Recall Across Sessions

External stores that survive beyond a single conversation — vector DBs, knowledge graphs, KV stores.

On this page
What it isWhere to store itWhat to storeBringing it backQuiz
Definition

What long-term memory is

Long-term memory is everything the agent knows about a user or context that survives beyond a single session. It is stored externally — in a database, not in the model's context — and retrieved on demand.

✓ What gets stored

  • User preferences and facts
  • Past decisions and outcomes
  • Domain knowledge (docs, FAQs)
  • Conversation summaries
  • Learned user patterns

✗ What should NOT be stored

  • Raw PII (credit cards, SSNs)
  • Contradicted outdated facts
  • Low-confidence extractions
  • Ephemeral small talk
  • Duplicate facts
Storage Options

Where to store it

1

Vector database

Best for semantic recall. Store fact embeddings, retrieve by cosine similarity. Examples: Pinecone, Weaviate, pgvector, Qdrant. See Chapter 08c.

2

Key-value store

Best for structured, frequently-accessed facts. "user:priya:location" → "Berlin". Millisecond reads. Redis, DynamoDB, or a simple SQL row per user.

3

Knowledge graph

Best for relational facts. "Priya works_at CompanyX which is_in Berlin." Multi-hop reasoning. See Chapter 08d.

4

Relational DB

Best for structured episodic logs. Full transcripts, timestamped events, easy date-range queries.

The Write Decision

What to store

⚠ Quality over quantity

Whatever you store, you will retrieve. Garbage in, garbage out. Store extracted facts, not raw turns. Storing the sentence "User said they live in Istanbul" is worse than storing the clean fact "User lives in Istanbul."

Retrieval

Bringing it back into working memory

Long-term memory only becomes useful when retrieved and injected into the current prompt.

retrieval.py
def build_prompt(user_message: str, user_id: str) -> str:
    # 1. Retrieve relevant long-term memories
    query_vec = embed(user_message)
    memories = vector_db.search(
        vector=query_vec,
        filter={"user_id": user_id},
        top_k=5
    )
    # 2. Assemble into prompt
    memory_block = "\n".join(f"- {m.text}" for m in memories)
    return f"[Memory]\n{memory_block}\n\n[User]\n{user_message}"
When is a vector database the best storage option for long-term memory?
AWhen you need millisecond lookup by exact key
BWhen you need semantic recall — finding memories by meaning not exact match
CWhen you need to store relational facts with multi-hop reasoning
DFor all memory types equally
