Vector Store: The Production Standard

Embed every memory, retrieve top-k by similarity. The default for RAG-shaped agents and the foundation of most production systems.

On this page

Definition

What a vector store architecture is

The vector store architecture embeds every memory as a vector and stores it in a dedicated vector database. At retrieval time, the query is embedded and the k nearest memories are returned.

This is the architecture described throughout Chapters 04–06. If you have read those chapters, you already understand how this works — this chapter ties it together as a complete memory system.

The Full System

How it works end-to-end

1

Write path

Extract a fact from the conversation turn. PII check it. Embed it. Upsert into the vector DB with user metadata.

2

Read path

Embed the query. Search the vector DB filtered by user_id. Return top-k. Inject into the prompt.

3

Update path

When a fact changes, search for the old fact by similarity. Mark it superseded. Write the new fact with a pointer to the old one.

4

Forget path

Explicit deletion request or TTL expiry. Delete the vector and all derived indexes. Log the deletion event.

vector_memory.py
class VectorMemory:
    def __init__(self, vector_db, embed_fn, pii_filter):
        self.db = vector_db
        self.embed = embed_fn
        self.pii = pii_filter

    def write(self, fact: str, user_id: str):
        if self.pii.detect(fact):
            fact = self.pii.redact(fact)
        vec = self.embed(fact)
        self.db.upsert({"vector": vec, "text": fact, "user_id": user_id,
                         "status": "active", "created_at": utcnow()})

    def read(self, query: str, user_id: str, top_k=5):
        query_vec = self.embed(query)
        return self.db.search(query_vec, filter={"user_id": user_id,
                                                   "status": "active"}, top_k=top_k)
Tradeoffs

Pros and cons

✓ Advantages

  • Effectively unlimited capacity
  • Semantic recall across paraphrases
  • Mature tooling (Pinecone, Qdrant, pgvector)
  • Scales to billions of memories
  • Filters on arbitrary metadata

✗ Disadvantages

  • Retrieval quality depends on embedding model
  • No native notion of relations or time
  • Infrastructure to operate
  • Easy to poison if writes are unguarded
  • Changing embedding model requires re-embedding
💡 Most agents need this

If you are building any agent with cross-session memory, a vector store is almost always the right choice. Start here.

Adding Governance

Making it production-ready

⚠ A raw vector store is not a memory system

A vector store handles storage and retrieval. A memory system adds governance: PII filtering, conflict detection, supersession, audit logging, permission scoping, and decay. You need both.

1

PII filter

Run a PII detection pass before writing any fact. Redact or block sensitive data.

2

Conflict detection

Search for similar existing facts before writing. Mark conflicting ones as superseded.

3

Permission scoping

Every read query must filter by user_id (and optionally tenant_id). Never expose cross-user data.

4

Audit logging

Every write, update, and delete is logged with source, timestamp, and actor.
