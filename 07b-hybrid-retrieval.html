Hybrid Retrieval: Dense + Sparse + Graph

No single retriever is optimal for all queries. Running all three and fusing the results is how production systems achieve high recall.

On this page
Why three retrievers?Dense retrievalSparse retrievalGraph retrievalFusion and rerankingQuiz
Motivation

Why three retrievers?

Think of three specialists, each excellent at different things:

Dense

Expert at meaning. Finds "hiking" when you ask about "outdoor activities."

Sparse (BM25)

Expert at exact text. Finds "order #12345" or rare technical terms.

Graph

Expert at relationships. Finds "user's employer's address" via entity links.

You do not pick one specialist. You ask all three and let them vote.

Dense Retrieval

Embedding-based semantic search

Dense retrieval embeds the query and searches the vector store for the nearest neighbors by cosine similarity. This is what Chapters 04–06 covered in depth.

StrengthsWeaknesses
Semantic similarity — finds paraphrasesPoor on rare tokens, IDs, codes
Handles natural language wellDoes not support boolean queries
Works across languages (multilingual models)Requires embedding model at query time
💡 When dense alone is enough

Dense retrieval handles the majority of real user queries well. Add sparse and graph only when your evaluation shows recall gaps on exact-match or relational queries.

Sparse Retrieval

BM25 keyword matching

BM25 (Best Match 25) is a probabilistic ranking function based on term frequency and inverse document frequency. It finds documents that contain the exact query terms, weighted by how rare those terms are.

bm25.py
from rank_bm25 import BM25Okapi

# Index: tokenize all memory texts
corpus = [m.text.lower().split() for m in memory_store]
bm25 = BM25Okapi(corpus)

# Query
query_tokens = "order 12345".lower().split()
scores = bm25.get_scores(query_tokens)
top_k = sorted(enumerate(scores), key=lambda x: x[1], reverse=True)[:5]
⚠ When sparse saves you

If a user asks "what happened to project Athena?" and the word "Athena" appears in exactly one memory, dense retrieval might miss it because the embedding of "Athena" is far from generic project descriptions. BM25 finds it instantly because it looks for the exact word.

Graph Retrieval

Entity relationship traversal

A knowledge graph stores entities (people, places, organizations, products) and their relationships as nodes and edges. Graph retrieval traverses these edges to find connected information that vector search cannot.

graph.py
# Example: finding user's employer's city
query = "Where is the user's company headquartered?"

# Entity extraction
user_node = graph.get("user:priya_123")
# user_node --[works_at]--> studio_node
studio = user_node.follow("works_at")
# studio_node --[located_in]--> city_node
city = studio.follow("located_in")
print(city.name)  # "Los Angeles"

# This is impossible with pure vector search
# because the fact is not stored as one sentence
💡 Graph is optional

Graph retrieval requires significant infrastructure: an entity extraction pipeline, a graph database (Neo4j, Neptune), and relationship schemas. Only add it when you have clear multi-hop reasoning requirements.

Bringing It Together

Fusion and reranking

After running all three retrievers in parallel, results are fused with RRF (see Chapter 06b) and optionally reranked with a cross-encoder for final precision.

hybrid.py
import concurrent.futures

def hybrid_retrieve(query, memory_store, graph_db, k=5):
    with concurrent.futures.ThreadPoolExecutor() as ex:
        dense_f = ex.submit(dense_search, query, memory_store)
        sparse_f = ex.submit(sparse_search, query, memory_store)
        graph_f = ex.submit(graph_search, query, graph_db)
        dense, sparse, graph = dense_f.result(), sparse_f.result(), graph_f.result()

    fused = rrf_fuse([dense, sparse, graph])
    return rerank(query, fused[:50])[:k]
💡 Parallelism is key

Run all three retrievers in parallel. Sequential execution would add their latencies. Parallel execution costs only the slowest one.
