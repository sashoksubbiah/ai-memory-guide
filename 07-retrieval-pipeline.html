Retrieval is a Pipeline, Not a Lookup

Production retrieval runs many stages on every turn. Skipping any one of them is usually how things break.

On this page

The Full Pipeline

All the stages

1

Need detection

Should we hit the memory store at all? Some queries do not need retrieval.

2

Query rewrite

Rephrase the query for better embedding or keyword match.

3

Dense retrieval

Embed the (rewritten) query and search the vector store.

4

Sparse retrieval

Run BM25 or TF-IDF for exact keyword matches.

5

Graph walk

Traverse entity relationships for structured queries.

6

Fusion (RRF)

Combine results from all three retrievers into one ranked list.

7

Reranking

Run a cross-encoder reranker on the top-N candidates for more precise ordering.

8

Filter

Apply permission scope and temporal validity filters.

9

Pack

Select the top memories that fit in the token budget and inject into the prompt.

Filters

Permission scope and temporal validity

After fusion and reranking, two filters must run:

🔒 Permission scope

In multi-user or multi-agent systems, a memory may exist that the current user or agent is not allowed to see. Scope filters enforce this. Always run these before exposing retrieved content.

⏰ Temporal validity

A memory may be factually outdated: "User lives in Istanbul" when they moved to Berlin six months ago. Status filters remove superseded facts or flag them as historical.

⚠ Filters must run server-side

Never trust the LLM to ignore out-of-scope memories. Apply filters in the retrieval layer, before the prompt is assembled.

Reranking

Why rerank after fusion?

Bi-encoder retrieval (embedding similarity) is fast but imprecise — it compares query and document independently. Cross-encoder reranking compares them together, giving the model full attention across both. This is much more accurate but much slower — so you run it only on the top 20–50 candidates from fusion, not the entire store.

rerank.py
# After RRF fusion gives you top-50 candidates:
candidates = rrf_fuse(dense_results, sparse_results, graph_results)[:50]

# Rerank with a cross-encoder (much more accurate, runs on small candidate set)
scored = [
    (cross_encoder.predict(query, candidate.text), candidate)
    for candidate in candidates
]
final_results = [c for _, c in sorted(scored, reverse=True)][:5]
The Interesting Bugs

What actually breaks in production

The retrieval pipeline fails in subtle ways that are hard to debug:

BugDescriptionFix
Correct but staleRetrieved memory is technically correct but outdated. Agent confidently states old fact.Add temporal validity filter; decay confidence over time
Correct but wrong scopeRetrieved memory belongs to a different user. Returned due to missing permission filter.Tenant-scoped metadata filters on every query
Contradicting rowsTwo retrieved memories conflict. Agent averages or picks one arbitrarily.Conflict detection at write time; mark superseded facts
Retrieval poisoningA memory was written by a bad agent. It poisons the prompt for every subsequent turn.Write governance; source tracking; human review queues
