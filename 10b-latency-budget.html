The Latency Budget: Where Your p95 Goes

Every millisecond on the retrieval path shows up in user-perceived latency. Here is how to measure and optimize each stage.

The Goal

A realistic p95 budget

Target: under 800ms total retrieval latency at p95. Anything beyond that is felt by users as slowness. Here is where those milliseconds actually go:

StageTypical p95Optimization lever
Need detection (fast check)5–20msRule-based; no LLM call needed for simple cases
Query rewrite (optional LLM call)60–150msSkip for simple queries; use a fast small model
Dense embed + vector search20–80msCache embeddings; tune FAISS index params; shard at scale
BM25 sparse search5–30msPre-built inverted index; extremely fast
Graph walk (optional)30–100msLimit hop depth; cache frequent paths
RRF fusion1–5msPure CPU compute; negligible
Cross-encoder rerank100–300msThe biggest cost; skip on simple queries; cache top-N
Permission filter1–5msIndex on user_id; in-memory permission cache
Context packing5–20msPre-tokenize memories; cache token counts
The Biggest Wins

Where to optimize first

1

Skip retrieval entirely (need detection)

Chapter 07a. Saves 100% of retrieval cost on eligible queries. Often 20–40% of all user queries do not need memory retrieval at all.

2

Cache top facts per user in a KV store

Keep the top 10–20 most-accessed facts per user in Redis. Sub-millisecond reads, no vector search required. Typically covers 60–70% of retrieval reads.

3

Run retrievers in parallel, not sequentially

Dense, sparse, and graph retrievers run simultaneously. Total cost is the slowest retriever, not their sum.

4

Skip reranking on confident retrievals

Cross-encoder reranking adds 100–300ms. A retrieval confidence score can gate whether reranking is needed for a given query.

5

Reduce k

Fewer retrieved memories means less packing overhead. Top-3 is often as good as top-10 for most queries. Benchmark before increasing k.

KV Cache Pattern

Serving hot facts without vector search

kv_cache.py
import redis, json
r = redis.Redis()

def get_top_facts(user_id: str, top_k=10) -> list[str]:
    key = f"hot_facts:{user_id}"
    cached = r.get(key)
    if cached:
        return json.loads(cached)  # ~0.3ms

    # Cache miss: full vector search
    facts = vector_db.search(user_id=user_id, top_k=top_k)
    texts = [f.text for f in facts]

    # Cache 15 minutes
    r.setex(key, 900, json.dumps(texts))

    # Invalidate on write
    # (called from write path when new fact is stored)
    return texts

def invalidate_hot_cache(user_id: str):
    r.delete(f"hot_facts:{user_id}")
💡 Cache invalidation

Invalidate the hot cache whenever a new fact is written for this user. This is cheap (one Redis delete) and keeps the cache fresh within one write-path cycle.
