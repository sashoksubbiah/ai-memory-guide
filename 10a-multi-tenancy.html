Multi-Tenancy: Three Isolation Strategies

Namespace-per-tenant, single collection with filters, or tiered hybrid — the tradeoffs that decide your SLA and cost structure.

On this page
Why isolation mattersNamespace per tenantSingle collection + filterTiered hybridChoosingQuiz
Stakes

Why isolation matters

In a multi-tenant memory system, a single isolation failure exposes User A's memories to User B. This is a catastrophic privacy violation. The strategy you choose determines your blast radius when things go wrong.

⚠ The failure mode

Missing a tenant_id filter on one code path is enough to leak data across tenants. Isolation strategies differ in how much they rely on code correctness vs. infrastructure separation.

Strategy 1

Namespace per tenant

Each tenant gets a completely separate collection in the vector database. No cross-tenant data even at the storage layer.

AspectDetail
IsolationStrongest — physical separation at the storage level
Cost at scaleHigh — many collections add overhead in most vector DBs
MigrationSimple — move one collection to migrate a tenant
OperationsOne collection to manage per tenant
Best for<1,000 tenants, enterprise with strict data residency requirements
✓ Recommended for high-security deployments

When a tenant has strict compliance requirements (HIPAA, SOC2, GDPR with data residency), namespace isolation is the only defensible choice.

Strategy 2

Single collection with payload filter

All tenants share one collection. Isolation is enforced by filtering on tenant_id in every query.

AspectDetail
IsolationRelies entirely on query filter correctness
CostLowest — one collection for all tenants
RiskOne missed filter in any code path = data leakage across all tenants
Best forNon-sensitive data, internal tools, small teams with tight code review
⚠ This strategy is fragile

The isolation guarantee rests on every single query including the correct filter. One developer omitting the filter in a new code path breaks isolation for all tenants at once.

Strategy 3

Tiered hybrid

Small tenants share a collection with filters. Large enterprise tenants get dedicated namespaces. A routing layer directs each query to the correct store based on tenant configuration.

AspectDetail
IsolationStrong for enterprise tenants, filter-based for SMB tenants
CostBalanced — shared infrastructure for small tenants
OperationsTwo code paths to maintain: namespace and filter
Best forMost production SaaS with a mix of SMB and enterprise customers
routing.py
def get_collection(tenant_id: str) -> Collection:
    tenant = tenant_config[tenant_id]
    if tenant.tier == "enterprise":
        # Dedicated namespace: no filter needed
        return vector_db.collection(f"tenant_{tenant_id}")
    else:
        # Shared collection: MUST include filter on every query
        return SharedCollection(filter={"tenant_id": tenant_id})
Decision Guide

How to choose

1

Start with namespace-per-tenant if <1,000 tenants

The operational overhead is manageable and the isolation is the strongest. No risk of filter bugs leaking data.

2

Move to tiered hybrid at scale

When the overhead of thousands of collections becomes prohibitive, route large enterprise tenants to dedicated namespaces and small ones to a shared collection.

3

Never use filter-only for sensitive data in production

The code-correctness dependency is too fragile for healthcare, finance, or legal data.
