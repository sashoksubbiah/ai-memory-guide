Six Ways Shared Memory Hurts

Cross-user leakage, poison propagation, attribution loss — the failure modes every multi-agent system encounters.

The Patterns

Why these failures repeat

Multi-agent memory failures are predictable consequences of common design decisions. Understanding the six patterns helps you avoid them from the start.

Deep Dive

Cross-user leakage and poison propagation

Mode 1

Cross-user leakage

Researcher stores "user prefers vegan recipes" in project memory. Next session, a different user gets vegan suggestions.

Root cause

Missing user_id / tenant_id filter on reads. Scope too broad.

Prevention

user_id + tenant_id on every read. Default deny across tenants. Never expose a memory to a user it was not written by.

Mode 3

Poison propagation

One agent hallucinates a fact and writes it to shared scope. All other agents now use that incorrect fact as ground truth.

Root cause

No confidence threshold on shared writes. No source tracking. No human review queue.

Prevention

Confidence threshold for shared writes. Source tracking on every memory. Periodic human review of high-influence shared facts.
