Knowledge Graph: Structured Relational Memory

When facts have clear structure and relationships, a graph beats a vector store for multi-hop reasoning.

On this page
What it isGraph data modelGraph queriesTradeoffsWhen to add itQuiz
Definition

What a knowledge graph is

A knowledge graph stores entities (people, places, organizations, concepts) as nodes and their relationships as labeled edges. Instead of a sentence "Priya works at Refik Anadol Studio which is in LA", you store three connected nodes: Priya →[works_at]→ Studio →[located_in]→ Los Angeles.

💡 Multi-hop reasoning

A vector store cannot answer "Where is the user's employer headquartered?" without storing that as one sentence. A knowledge graph answers it by traversing user → employer → location — even if those facts were written separately.

Data Model

How entities and relationships are stored

graph_model.py
# Example: storing user knowledge as a graph
graph = [
    # Entities
    Node("user:priya", type="Person", name="Priya"),
    Node("org:refik_studio", type="Organization", name="Refik Anadol Studio"),
    Node("loc:los_angeles", type="Location", name="Los Angeles"),
    Node("skill:python", type="Skill", name="Python"),

    # Relationships
    Edge("user:priya", "works_at", "org:refik_studio", since="2023"),
    Edge("org:refik_studio", "located_in", "loc:los_angeles"),
    Edge("user:priya", "has_skill", "skill:python", proficiency="expert"),
]
⚠ Entity extraction is hard

To populate a knowledge graph, you must reliably extract entities and relationships from natural language. This requires a dedicated extraction pipeline (usually another LLM call). The quality of the graph depends entirely on the quality of extraction.

Graph Queries

How to query a knowledge graph

graph_queries.py
# Cypher (Neo4j query language)

# "Where does the user's employer operate?"
MATCH (u:Person {id: 'user:priya'})
      -[:works_at]->(org:Organization)
      -[:located_in]->(loc:Location)
RETURN loc.name

# "Who does the user work with who knows machine learning?"
MATCH (u:Person {id: 'user:priya'})
      -[:works_at]->(org:Organization)
      <-[:works_at]-(colleague:Person)
      -[:has_skill]->(skill:Skill {name: 'machine learning'})
RETURN colleague.name
Tradeoffs

Pros and cons

✓ Advantages

  • Multi-hop relational reasoning
  • Native supersession (mark edges as expired)
  • Precise structured queries
  • Clear audit trail per relationship

✗ Disadvantages

  • Complex extraction pipeline required
  • Requires graph database infrastructure
  • Schema design is hard to change later
  • Poor at unstructured text similarity
When to Add It

Choosing a knowledge graph

✓ Add knowledge graph when

You have clear relational queries (multi-hop). Your domain has well-defined entities (users, orgs, products). You need precise structured answers. Your team can build and maintain an extraction pipeline.

✗ Skip if

Most queries are freeform text similarity. Your entity schema is unclear. You cannot build a reliable extraction pipeline. Vector store recall is already good enough.

What query type benefits most from a knowledge graph over a vector store?
AFinding memories with similar meaning to a query phrase
BMulti-hop relational queries like "What city does the user's employer operate in?"
