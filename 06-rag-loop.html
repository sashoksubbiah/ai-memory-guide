Overview

What RAG is

RAG — Retrieval-Augmented Generation — is the pattern most production agents use on every turn. Instead of relying solely on the model's training data, the agent retrieves relevant information at query time and includes it in the prompt.

💡 Why this matters

Without RAG, the model only knows what it learned during training. With RAG, the model can access user-specific facts, recent information, and private knowledge — all dynamically at query time.

The Pipeline

The 8-step RAG loop

1

User query arrives

A fresh user message contains both a question and possibly a new fact about the user.

2

Embed the query

The query is converted to a vector using the same embedding model used for storing memories.

3

Search the store

The query vector is used to search the vector database, returning candidate memories.

4

Retrieve top-k

The k most similar memories are returned. k is typically 3–7.

5

Compose the prompt

The retrieved memories, system prompt, and conversation history are assembled into one prompt.

6

LLM generates

The assembled prompt is sent to the LLM. It generates a response with full access to the retrieved context.

7

Govern new info

The agent decides whether the user's message contains any new facts worth saving.

8

Update the store

New facts are extracted, governed (PII check, conflict check), and written to the memory store.

💡 Every stage is a budget

Each stage in the loop is a place where you can spend engineering effort: better embeddings, smarter retrieval, prompt compression, stricter governance. They are all the same pipeline with more care taken at different steps.

The Write Side

Completing the loop

⚠ The write path is as important as the read path

Many teams build a great retrieval pipeline and then write memories back naively. Poor write quality — storing raw turns, skipping governance, allowing PII — degrades retrieval precision over time. Garbage in, garbage out.

full_rag.py
def run_rag_turn(user_msg, user_id, memory_store):
    # === READ PATH ===
    query_vec = embed(user_msg)
    retrieved = memory_store.search(query_vec, user_id=user_id, top_k=5)
    prompt = build_prompt(system_prompt, retrieved, user_msg)
    response = llm.call(prompt)

    # === WRITE PATH ===
    extraction = llm.call(EXTRACT_PROMPT.format(turn=user_msg))
    facts = json.loads(extraction).get("facts", [])
    for fact in facts:
        governed_write(fact, memory_store, user_id=user_id)

    return response
Production Tricks

HyDE and RRF

Two techniques that significantly improve RAG quality in production:

HyDE

Questions and answers are different shapes in embedding space. HyDE generates a hypothetical answer first, embeds that, then searches. The fake answer does not need to be correct — just shaped like the target.

Deep dive →

Reciprocal Rank Fusion

Run multiple retrievers (dense + sparse + graph) in parallel. Fuse their ranked lists using RRF. The combined result beats any single retriever on hard queries.

💡 Start simple

Get plain vector search working end-to-end first. Add HyDE and RRF when you can measure that they improve your specific recall metrics.
